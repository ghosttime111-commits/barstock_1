# PWA в BarStock

BarStock устанавливается как PWA, но остаётся серверным приложением TanStack Start.
Service worker не превращает приложение в SPA и не хранит рабочие данные офлайн.

## Что кешируется

Во время production-сборки Workbox добавляет в precache только публичные ресурсы:

- хешированные JavaScript- и CSS-файлы;
- PWA-иконки, favicon и Apple Touch Icon;
- локальные шрифты и статические изображения;
- `manifest.webmanifest`;
- статичная страница `offline.html`.

SSR-навигация использует стратегию network first без кеширования ответа. Если сеть
недоступна, service worker возвращает только обезличенную `offline.html`.

## Что не кешируется

Service worker не создаёт runtime-кэши для API и не перехватывает не-GET запросы.
В частности, не кешируются:

- POST-запросы и TanStack server functions;
- Supabase API;
- вход, сессии и списки пользователей;
- переучёты, отчёты, списания, перемещения и сообщения;
- JSON с рабочими данными;
- Excel-файлы;
- SSR HTML с данными пользователя.

## Локальная проверка

```powershell
npm install
npm test
npx tsc --noEmit
npm run lint
npm run build
npm run preview -- --host 127.0.0.1
```

PWA включена только в production-сборке. Для полной проверки SSR-развёртывания
используйте Vercel Preview Deployment. На `localhost` браузеры также разрешают
service worker без HTTPS.

В DevTools проверьте:

1. Application > Manifest: имя, иконки 192/512/maskable, `start_url: /`,
   `display: standalone`.
2. Application > Service Workers: worker активирован со scope `/`.
3. Cache Storage: только versioned assets и публичные PWA-файлы.
4. Network: server functions и POST-запросы всегда идут в сеть.

## Установка

В Chrome и Edge пункт `Установить BarStock` появляется после события
`beforeinstallprompt` и скрывается после установки. На iPhone/iPad вместо
программного prompt показывается инструкция через меню «Поделиться».

Установленное приложение запускается в режиме `standalone`. Manifest shortcuts
ведут на обычные маршруты, поэтому существующая авторизация и RBAC продолжают
проверять доступ.

## Проверка offline

1. Откройте production или Vercel Preview онлайн и дождитесь активации worker.
2. В DevTools включите Offline.
3. Перезагрузите любой внутренний URL.
4. Должна открыться обезличенная страница «Нет подключения к интернету».
5. Верните сеть и нажмите `Повторить`.

Индикатор `navigator.onLine` служит только подсказкой и не блокирует интерфейс.

## Обновление

Новый worker ожидает подтверждения пользователя. BarStock показывает сообщение
`Доступна новая версия BarStock` с действиями `Обновить` и `Позже`.

Автоматической перезагрузки нет. Перед обновлением проверяется общий признак
несохранённых изменений. Если он установлен, приложение дополнительно
предупреждает о перезагрузке. После подтверждения worker получает
`SKIP_WAITING`, активируется и страница перезагружается один раз.

## Очистка в разработке

Chrome/Edge DevTools:

1. Application > Service Workers > Unregister.
2. Application > Storage > Clear site data.
3. Закройте остальные вкладки BarStock и перезагрузите страницу.

Это полезно после изменения стратегии кеширования или manifest.

## Ограничения первой версии

- Рабочие данные нельзя создавать или изменять без сети.
- Нет очереди POST-запросов, Background Sync и кеша Supabase.
- Нет push-уведомлений.
- `navigator.onLine` не гарантирует доступность сервера.
- Установку и обновление нужно дополнительно проверять на целевых Android,
  Windows и iOS устройствах.

Полноценный offline-режим потребует отдельного защищённого локального хранилища,
конфликт-резолюции, очереди команд и серверной идемпотентности. Его нельзя
добавлять простым расширением текущего precache.
